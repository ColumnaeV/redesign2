DESIGN

Drivetrain

We decided to go with the H drive because this year the holonomic drive is going to be not as effective as in past years. The drive would have problems going over the bar to get to the 20 point zone. We need to make our drivetrain with the ability to get over the bar easily. With this in mind we added a chain loop in between the wheels. This chain is not to connect the two wheels as it is help the robot get over the bar. When the front two wheels go over the bar, and not be able to touch it anymore, the chain will still be touching the bar with enough pressure to act the same as tank treads. The chain will still move the robot even when the wheels aren’t touching. And with this method we won’t have to sacrifice the downfalls of tank treads (low mobility, high friction, and they catch on more things than normal).

Lift / Scoring

This year we have two main ideas: a reverse four bar and 6 bar lift. The 6 bar lift would be easier to build but not as effective as the reverse four bar lift. The reverse four bar lift is the better lift because it will extend up more that up and out (which is what the 6 bar does). Both of the real sketches are in this folder but we put smaller versions here--->. We as a group have decided to try the reverse four bar lift unless it doesn’t get done in time in which case we will build the 6 bar lift for competition.

Autonomous Goals

In the autonomous this year we have two goals: score as many points as possible and to set ourselves up to be able to score fast in the main round. As for setting ourselves up we want to stack a few cones on a mobile goal so we can easily score it come the driver control period.

Code

Currently we are using robotc as our ide for compiling and downloading our code to the robot. We have a autonomous selection program the make use of our lcd screen in order for us to make rapid changes of the currently selected autonomous code. We have also made a function that takes a the normal normal voltage per second graph and alters the amount of time that it would take in order for the motors to reach full rotational speed. This is know as a slew rate. Simply it makes the motor accelerate at a slower rate at the beginning and end of the movement to prevent jerking of the motors. We have also used the lcd screen during the drive period to display a debug array, so that we can determine the reason why we had a failure or a loose in drive or lift control or any number of different factors.

WEB DEV SECTION

The Development

This website was developed in order to demonstrate the capability of a group of students on a Millard North robotics team to create a digital documentation method for their robot. The students at Millard North are held to the standards to display a student’s ingenuity and creativity through any type of medium without the support of professional help or pre-built templates such as wix or google sites. With that being said, we have taken it upon ourselves to design, construct, and host our own website.

Over the period of a couple of months, this project has gone through four main builds as well as a multitude of version updates and alpha renditions. In the very beginning of this project, the leading factor to the development of this was to only be used as the notebook for the robot’s construction progression. The initial intention was to find an easier and more eco-friendly method to present our data while visually pleasing to the judges. This idea was taken from a Millard North team from a couple of years that found it to be a success. From then on, I began to write more code to support other ideas that led to dead ends. Within the course of two months, the project had went through two complete, dead-end builds and onto the third build. With too many changes happening within one version update, the third build was eventually scrapped and downsized for the fourth and current build.

In this current build, I drew from previous projects and builds and optimised each section to its best ability and functionality to the user. Written to be supported on all desktop mobile viewports, accessible navigation, ease on eyes for any time of the day with light and dark modes, and quality design are the main highlights in which makes this website superior to pre-templated websites.

Along with that we have created the infrastructure for which our website is being hosted from. To understand how the back end of the website was made, you first have to understand how a website server works. First of the there are many different types of transmission control protocols out there but for a web server only HTTP port 80 and HTTPS port 443 is needed. HTTP stands for hypertext transmission protocol which in laymans terms means it transfers a HTML based website from the server to the client using get and post request. HTTPS (which is what our website is using) on the other hand is HTTP with a secure sockets layer which authenticates you with the server to keep the data transfer encrypted.

Now that you know the different types we can move on to our website server. We are using a Windows based website server using Internet Information Services (IIS). This a graphic user interface based server which makes rapid changes to the server fast and simple. To start out we pointed our server to our local copy of the website. This is the server to know what file is needed for the website. Next, we move on to changing the default documentation page to have index.html first as it is the home page on our website. From there we bounded our ip to the domain name which is www.mnhsrobotics.com on port 80 and 443. Next we configured the URL rewrite feature to rewrite our HTTP requests to HTTPS requests. This is why even when you type in http://www.mnhsrobotics.com it redirects to https://www.mnhsrobotics.com. Finally we had to buy and point our domain name to the server's IP address.
